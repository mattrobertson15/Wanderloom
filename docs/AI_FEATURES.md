# AI Features

AI is light and optional at MVP. No core flow (creating a trip, post, or photo) should ever require an AI call to succeed.

## MVP AI Helpers

1. **Caption suggestions** — given a post's photos (and optionally place name), suggest 1-3 short caption options the user can accept, edit, or ignore.
2. **Tag suggestions** — suggest descriptive tags/keywords for a post based on its text/place, optional metadata for future search/discovery.
3. **Trip summaries** — generate a short blurb summarizing a trip from its posts' text and place names, for use on the trip cover/public page.
4. **Public guide generation** — turn a trip into a shareable "guide"-style page (places visited, short narrative) generated from existing post content. If a full LLM integration isn't ready in a given session, this can ship as a deterministic template (e.g. ordered list of places + post excerpts) and be upgraded later without changing the data contract.

All of the above are invoked on-demand (a button the user clicks), never automatically on save, to keep cost and latency off the critical path.

## Roadmap AI Features

- **EXIF place extraction** — read GPS/timestamp EXIF data from uploaded photos to pre-fill `taken_at`/`location` and suggest a Place.
- **Photo clustering by location/date** — group uploaded photos into suggested Posts/Albums automatically.
- **Auto-generated trip timeline** — reconstruct a chronological itinerary view from posts/photos.
- **Itinerary generation from saved places** — given a user's saved places (from browsing), draft a suggested itinerary/trip skeleton.
- **Recommendations from friends' maps** — surface places/trips from a user's social graph relevant to a destination they're planning.
- **Chat with travel memories** — conversational interface over a user's own trip history ("when did we go to that beach in Portugal?").

## Data Needed for AI

- MVP helpers need: post text, post photos (URLs), attached place name/category, trip title/date range. All already present in the core schema — no extra AI-specific tables required at MVP.
- Roadmap features will need: raw EXIF metadata (currently not extracted/stored), richer place metadata (category, popularity), and aggregated cross-user signals (for recommendations) — these are intentionally deferred schema additions, not MVP scope.

## Prompting Assumptions

- Treat all AI features as thin wrappers around a single provider-agnostic interface in `packages/domain` (or a future `packages/ai`) so the underlying model/provider can change without touching UI code.
- Prompts should be constructed server-side (Next.js route handler / server action, or a Supabase Edge Function) — never call a model provider directly from client code, to avoid exposing API keys and to allow rate limiting.
- Keep prompts narrowly scoped per feature (one prompt template per helper) rather than a general-purpose "do anything" assistant at MVP.

## Safety/Privacy Considerations

- Only send content the requesting user already owns or has been granted access to view to any AI provider — never send another user's private content as context.
- Public guide generation should only operate on posts whose effective visibility is already `public` (or the trip is being generated by its own owner, regardless of current visibility, since they already have full access).
- Do not send PII beyond what's already user-authored/public-facing (e.g. don't construct prompts that include email addresses or auth identifiers).
- Make AI suggestions clearly editable/dismissable — never auto-publish AI-generated text without explicit user action.

## Cost Considerations

- All AI calls are on-demand and user-initiated, which naturally caps cost relative to usage.
- Track AI usage via the `events` table (`ai_usage` event type) from day one so cost can be monitored and rate-limited per user if needed, even though no user-facing quota exists at MVP.
- Favor smaller/cheaper models for short, well-scoped tasks (captions, tags) and reserve any larger-model usage for guide/summary generation where quality matters more.
